# ゲームサンプル実装メモ

`samples/maze_chase.rb` は、X68000 上の mruby でゲーム的な処理を試すためのパックマン風サンプルです。
元になった Ruby+Gosu 版の考え方を残しつつ、描画と入力を X68000 の BG / スプライト / ジョイスティックへ置き換えています。

## 基本方針

- 迷路はBGで表示する
- 自機と敵はスプライトで表示する
- マップは8x8セル単位で管理する
- 移動オブジェクトは16x16相当として扱う
- Ruby側ではゲームロジックを書き、表示更新はできるだけBG/スプライトへ寄せる

## 通行判定

最初は通行可否テーブルを起動時に生成していましたが、起動が重くなる割に効果が薄かったためやめました。

現在は、移動先の2x2セルだけを `MAP_DATA` から直接確認しています。これにより、起動時間が短くなり、移動中のカクつきも避けられています。

## 速度

速度差は固定小数点で表現しています。

```ruby
FIX_ONE = 256
PLAYER_SPEED = 512       # 2.0 dot/frame
GHOST_SPEED = 384        # 1.5 dot/frame
GHOST_FRIGHT_SPEED = 256 # 1.0 dot/frame
GHOST_TUNNEL_SPEED = 256 # 1.0 dot/frame
```

整数ドット座標だけで速度差を作ると、1dot/2dotのように粗くなりすぎます。固定小数点にすると速度差を表現しやすくなります。

注意点として、8dotセル境界をまたぐ時に扱いを誤ると、壁判定のタイミングを飛び越えて壁へ入ることがあります。
右/下方向は境界をまたいだ時に8dot位置へ丸めていますが、左/上方向で同じ丸めを行うと逆に8dotジャンプになり、倍速に見えることがあります。
このため、方向ごとに丸めの扱いを分けています。

## 敵AI

敵は `GhostActor` クラスで管理しています。

- 通常時は自機や散開ターゲットを目指す
- パワー餌中は自機から遠ざかる
- 巣の中では出口へ向かう専用ルールを優先する
- 反撃中に食べられると巣へ戻る

完全な経路探索ではなく、軽量なターゲット方向選択にしています。MC68000 上では、毎フレームの重い探索よりも、セル中心での簡単な判断のほうが安定します。

## HUD

スコアや残機はBG/PCGで表示します。

グラフィック面へ小さな文字を `Graph.fill` で描く方式も試しましたが、描画呼び出しが多くなりカクつきの原因になりました。BGセルを書き換える方式なら、更新頻度が低く軽量です。

## 反撃モード

パワー餌取得時に一定フレームだけ反撃モードへ入ります。

- 敵は青いパターンへ切り替わる
- 終了直前は青/通常色で点滅する
- 反撃中の敵は自機から逃げる
- 捕まえるとスコア加算し、その敵だけ巣へ戻る

## VBlank待ち

ゲームループの最後で `X68k::Crtc.wait_vblank` を呼び、画面更新を垂直帰線期間に同期しています。
Ruby側の処理が軽い場合でも、同じ帰線期間中に複数回進んでしまうことを避けるため、`wait_vblank` は一度表示期間へ抜けてから次の垂直帰線期間を待つ実装にしています。

高速な実機やアクセラレータ環境では、毎VBlank更新だとゲーム速度が速すぎる場合があります。
その場合は `wait=2`〜`wait=4` のように起動引数でゲームループごとに待つVBlank数を増やします。
CPUループによるウェイトではなくVBlank数で調整するため、CPU速度の違いに引きずられにくくなります。

## ワープ通路

中央左右の抜け道は、左端から出ると右端へ、右端から出ると左端へ移動するワープ通路にしています。
ワープ通路の手前では通常状態の敵だけ速度を落とし、プレイヤーが逃げ切る余地を作っています。

## Z-MUSIC

`maze_chase.rb` は `samples/ZMD/` のサンプル音源を使います。
通常BGMとパワー餌中BGM、ミス時ジングルは `play_bgm` で再生し、短い効果音は `play_se` で鳴らしています。

Z-MUSIC のZMD効果音はBGMとテンポが共通になるため、テンポを変えるSEを鳴らすとBGM側にも影響します。
短いSEはテンポ変更を含まないZMDにするか、ADPCM再生と使い分けます。

## 分かったこと

X68000 + mruby でも、ゲームループ、クラス化、状態管理、入力処理は十分扱えます。

ただし、MC68000 ではRubyのオブジェクト生成や配列処理が重くなりやすいため、以下の方針が有効です。

- 毎フレームの配列生成を避ける
- 文字列比較を避け、マップは数値化する
- 表示はスプライト/BGを使う
- 細かい描画をRubyから連発しない
- 起動時の大きな生成処理は減らす
- 必要なら重い処理をC拡張へ移す
